# README Review Checklist

## 1. First View

- [ ] The H1 is the project name.
- [ ] One or two sentences explain what the project does.
- [ ] The reader value, the Why, appears before implementation detail.
- [ ] Core paths are visible: Quickstart, Docs, and Issues.

## 2. Quickstart

- [ ] `Requirements` are stated.
- [ ] `Install` is stated.
- [ ] `Run` or a minimal execution example is present.
- [ ] The steps are executable and avoid unnecessary branches.

## 3. Core Sections

- [ ] `Usage` is present.
- [ ] There is a path to `Documentation`.
- [ ] There is a path to `Getting help`, such as Issues or Discussions.
- [ ] There is a path to `Contributing`.
- [ ] `License` is stated.

## 4. Structure

- [ ] The README is limited to information needed for adoption.
- [ ] Long-form details are moved to `docs/`.
- [ ] Links prefer relative paths when possible.
- [ ] The heading structure is organized.

## 5. Quality

- [ ] There are no stale commands or broken links.
- [ ] Version assumptions are explicit.
- [ ] Terminology is consistent.
- [ ] Images and badges are not excessive.

## Severity Guide

### Critical

- The minimal run path does not work.
- A reader cannot decide whether to start from the README alone.

### Warning

- Information order makes value or setup hard to find.
- Links to docs are weak.

### Info

- Wording or examples could be improved.
